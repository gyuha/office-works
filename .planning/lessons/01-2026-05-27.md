## Lens: Five Whys
_Captured: 2026-05-27T23:00:02Z_

### Problem Statement
role_menu_permissions와 user_menu_permissions에서 menu_id 인덱스가 플랜에 누락되어 코드 리뷰 후 발견됨

### Why 1
플랜이 role_id 인덱스만 명시함

### Why 2
주 쿼리 패턴(findByRoleId)만 고려함 — Phase 2 서비스 코드가 사용할 직접 쿼리만 인덱싱 계획에 포함

### Why 3
DB 성능 관점보다 인터페이스 설계에 집중함 — FK CASCADE 경로 쿼리(메뉴 삭제 시 연관 권한 행 접근)를 고려하지 않음

### Why 4
Phase 1의 성공 기준이 컴파일 성공 + 요구사항 출력에 집중됨 — 인덱스 성능 항목이 Success Criteria에 없어 플랜 작성 시 편향 발생

### Why 5
DB 스키마 페이즈를 '구조 정의' 레이어로만 범위 제한하고 '성능 적정성 검증'은 다음 페이즈로 미룬다고 가정함

### Root Cause
스키마 페이즈 Success Criteria에 "FK가 있는 컬럼에 인덱스 생성 여부" 항목이 없어서, 플랜 작성자와 리뷰어 모두 기능적 정확성(컴파일, 요구사항 충족)에만 집중하고 인덱스 완전성은 검토하지 않게 됨

### Action Items
| priority | item | next step |
|----------|------|-----------|
| P1 | 스키마 페이즈 Success Criteria에 "모든 FK 컬럼에 인덱스 생성 여부" 항목 추가 | Phase 2 이후 스키마 작업 시 PLAN.md success_criteria에 명시 |
| P1 | PLAN.md 연결 테이블 섹션에 양방향 FK 인덱스를 기본 요건으로 포함 | PLAN 템플릿 또는 CONVENTIONS.md에 "FK 컬럼 → 인덱스 필수" 규칙 추가 |
| P2 | 코드 리뷰 프롬프트에 "인덱스 컴플리언스(FK 컬럼 인덱스 커버리지)" 체크 항목 포함 | 리뷰어 프롬프트 개선 |
