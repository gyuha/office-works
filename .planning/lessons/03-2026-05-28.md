### Decisions
- D-05: ServerAccessDeniedHandler 구현 + ApiResponse envelope 형식 403 반환 — Spring Security WebFlux 인터페이스 직접 구현 (03-CONTEXT.md)
- D-10/D-11: SecurityConfig에 `/api/menus/**` authenticated 명시 + Controller `@PreAuthorize("hasRole('ADMIN')")` 조합 — 두 레이어에서 각자 역할 분리 (03-CONTEXT.md)
- D-13: JwtAuthenticationFilter 변경 금지 — Phase 3 성공 기준 4번, 모든 플랜에서 준수 (03-01-PLAN.md, 03-02-PLAN.md)
- D-02: AdminMenuResponse는 canRead/canWrite 없는 5개 필드 — MyMenuResponse와 목적이 달라 별개 DTO (03-CONTEXT.md D-01~D-02)

### Surprises
- `AdminMenuResponse.isActive` record 컴포넌트명 Jackson 직렬화 버그 — Java record의 `isActive()` 접근자가 Boolean getter로 인식되어 JSON 키가 `"active"`로 직렬화됨. 코드 리뷰에서 발견, `@JsonProperty("isActive")` 추가 (`7d7827e`)
- `@PreAuthorize` AccessDeniedException 전파 경로 — Spring WebFlux에서 `@PreAuthorize` 거부는 ExceptionTranslationWebFilter가 아닌 `@RestControllerAdvice`로 전파됨. 코드 리뷰어 "dead code" 판단(confidence 95%)이 틀렸고 테스트가 즉시 검증
- Controller 아키텍처 위반 — PLAN.md에서 `menuRepository.findAll()` 직접 호출로 계획됨. Controller가 Repository를 직접 의존하는 레이어 위반이 코드 리뷰 [3]에서 발견 (`7d7827e` 수정)

### Patterns
- Wave 병렬 실행 (Phase 1/2/3 모두 적용) — wave:1 독립 플랜 병렬, wave:2 의존 플랜 직렬. 파일 충돌 없는 병렬화로 Phase 3도 안정 완료
- CONTEXT.md D-XX 사전 잠금 (Phase 1/2/3 누적) — D-01~D-20 문서화로 실행 에이전트가 재결정 없이 플랜 완수
- 단일 Fix 커밋 패턴 — 코드 리뷰 이후 발견된 이슈를 하나의 Fix 커밋으로 처리 (`7d7827e`)
- 리뷰어 판단 → 테스트 검증 사이클 — 리뷰어 제안 적용 후 즉시 테스트 실행으로 오판 확인 및 즉시 복원

### Mistakes
- AdminMenuResponse `@JsonProperty("isActive")` 누락 — Java record 컴포넌트명 `isActive`가 Jackson Boolean getter 패턴으로 직렬화됨. 코드 리뷰 [2]에서 발견, `7d7827e` 수정 (AdminMenuResponse.java)
- MenuController가 MenuRepository 직접 의존 — `controller → application/service → domain/repository` 아키텍처 레이어 위반. PLAN.md 설계 단계에서 레이어 검증 미실시. 코드 리뷰 [3]에서 발견, `7d7827e`에서 getAllMenus() 서비스 메서드 추가로 수정
- 코드 리뷰어 제안 무검증 적용 → 테스트 실패 → 복원 — handleAccessDeniedException 제거 후 403 테스트 실패. WebFlux @PreAuthorize 동작 특성 미확인 (Phase 3 실행 중 1회 발생)
