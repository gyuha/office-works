### Decisions
- D-09: roleIds 빈 체크 명시적 분기 — R2DBC `IN (emptyList)` 동작 불확실성 제거. `findByRoleIdIn()` 호출 스킵 패턴으로 확정 (02-CONTEXT.md)
- D-11: `Mono.zip(rolePermsMono, userPermsMono)` 병렬 실행 — roleIds 있을 때만 병렬, 없으면 Mono.just(emptyList) 즉시 반환 (02-CONTEXT.md D-11)
- D-13: Phase 2에서 MenuAuthorizationBean Bean 구현만, @PreAuthorize 실제 적용은 Phase 3 — Phase 경계 명확화 (02-CONTEXT.md)
- D-14/D-16: canRead/canWrite 단건 메서드를 서비스에 분리하고 Bean이 호출 — 단일 canAccess() 방식 거부 (02-CONTEXT.md)

### Surprises
- MenuAuthorizationBean.canRead/canWrite가 항상 isAdmin=false로 서비스 호출 — D-08(ADMIN bypass)이 서비스에서만 처리되어 Bean에서 재체크가 필요함을 설계에서 미포착. Phase 3 @PreAuthorize 연계 시 ADMIN 권한 전면 차단 사고로 이어질 수 있었음 (`b351100` 코드 리뷰 [1]에서 발견)
- IT 테스트 `withValidUserToken`과 `withUserNoRoles`가 동일 토큰 사용 — BeforeEach에서 역할 없는 계정을 생성하여 두 시나리오가 사실상 동일 (코드 리뷰 [4])

### Patterns
- CONTEXT.md D-XX 사전 잠금 패턴 (Phase 1에서 이어짐) — 실행 에이전트가 19개 결정 사전 확인 후 탐색 없이 구현 집중. 대규모 플랜 완주의 핵심 요인
- Wave 병렬 실행 (Phase 1에서 이어짐) — 의존성 없는 플랜을 wave:1로 동시 실행 + 의존성 있는 플랜을 wave:2/3으로 직렬화

- Java record DTO + 생성자 주입 전용 패턴 — 프로젝트 전체 일관성 유지

### Mistakes
- MenuAuthorizationBean ADMIN bypass 누락 — `canRead(userId, menuCode)` 서비스 메서드가 isAdmin 파라미터 없이 설계되어 Bean에서 ADMIN 체크 없이 직접 호출. 코드 리뷰 [1]에서 발견, `b351100`에서 `isAdmin()` 분기 추가 (`MenuAuthorizationBean.java:33`)
- 역할 없는 사용자 + user override 단위 테스트 누락 — D-10(user override는 항상 실행)을 반영한 시나리오가 테스트에 없었음. 코드 리뷰 [2] 발견, `b351100`에서 추가 (`MenuPermissionServiceTest` 8번째 테스트)
- ADMIN IT 테스트 canRead/canWrite 미검증 — DisplayName에 "canRead=true, canWrite=true로 반환"이라고 명시했지만 실제 assertion 없음. 코드 리뷰 [3] 발견, `b351100`에서 jsonPath 검증 추가
- IT 테스트 중복 시나리오 작성 — 역할 없는 동일 토큰으로 두 테스트 메서드 작성. 코드 리뷰 [4] 발견, `b351100`에서 중복 테스트 제거
